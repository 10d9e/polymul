## Approach

Fuse forward len=64 + len=16 stages into one radix-16 pass
The two strided radix-4 forward stages (half-blocks 64 and 16) each
round-tripped the full xab spectrum through linear memory. Fuse them into a
single radix-16 = radix-4(len=64) ∘ radix-4(len=16) pass: for each 256-block
and offset jj in 0..16, gather the 16 elements at stride 16, run both radix-4
sub-stages in registers, and scatter back — halving this section's memory
traffic (~1 store + 1 load per element per prime removed).
Two details make it a win rather than the regression a naive version gives:
  * The 16-element tile is held in sixteen explicit named locals, not a [L;16]
    array — under the strided gather/scatter LLVM keeps named locals in wasm
    locals (free), whereas the array fails SROA and spills to memory.
  * The loop is column(jj)-outer so each butterfly's five twiddle vectors are
    loaded once and reused across the four 256-blocks, matching the hoisting
    the separate butterfly-outer stages had.
Sub-stage A (len=64) twiddles are w^{4jj+64k} for group k; sub-stage B (len=16)
shares w^{16jj}. Output is bit-identical (same wasm checksums; native gate and
all 32 fixtures pass). WORK 8325888 -> 8009088.
